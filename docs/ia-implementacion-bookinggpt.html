IA en BookingGPT: backend y mobile

Arquitectura completa del sistema de IA: endpoints, seguridad, modelos, herramientas, motor semantico, busqueda hotelera, streaming SSE, integracion mobile y estado conversacional.

Entrada principal Mobile consume /api/ai/chat/stream via XHR con fallback a fetch. Web usa fetch directo (legacy).
Motor IA Dos llamadas a OpenAI: Call 1 elige herramienta (tool calling), Call 2 redacta respuesta. Motor semantico corre en paralelo.
Estado persistido Sesiones, mensajes, plan de busqueda, inventario mostrado, snapshots y feedback se guardan por usuario.
Idea central: el modelo no busca hoteles directamente. OpenAI decide la herramienta; el backend valida, ejecuta busquedas, aplica el motor semantico, resuelve lugares, maneja disponibilidad real, ordena carruseles y renderiza la respuesta para la app. Todas las decisiones sensibles de negocio permanecen en servidor.

1. Resumen Ejecutivo y Arquitectura

La IA esta centralizada bajo /api/ai. Mobile usa principalmente /api/ai/chat/stream via SSE. Existe una ruta legacy /api/assistant/search que internamente reutiliza el mismo controller de IA.

📱 Mobile (XHR → fetch fallback) → POST /api/ai/chat/stream → 🔒 Auth + Rate limit + Moderacion
🔒 Validacion OK → OpenAI Call 1: tool selection (gpt-5.4) → Backend ejecuta herramienta
🔍 Herramienta ejecutada → Motor semantico + Busqueda + Carruseles → OpenAI Call 2: redaccion (gpt-5-mini / gpt-5.4)
💬 Respuesta generada → SSE stream hacia mobile → Persistencia (sesion + snapshots)
Principio de diseno: el modelo de lenguaje actua como orquestador conversacional. Todas las decisiones de negocio (busqueda, filtros, disponibilidad, fallback geografico, cuotas, moderacion, semantica) se ejecutan en el backend, nunca se delegan al modelo.

Autenticacion y moderacion

Todos los endpoints requieren middleware authenticate. Si una sesion ya esta procesando un turno, la API devuelve 409 AI_CHAT_TURN_IN_PROGRESS. El lock expira automaticamente a los 90 segundos para evitar bloqueos permanentes.

Antes de invocar el modelo, el mensaje pasa por insiderBack/src/services/ai.contentModeration.service.js:

  • Fase 1 — regex local: detecta contenido ofensivo claro con patrones predefinidos. Es sincrono y sin latencia.
  • Fase 2 — OpenAI Moderation API: se invoca si OPENAI_API_KEY esta disponible. Si falla, el sistema no bloquea el mensaje (fail-open intencional para no degradar UX).

3. Flujo Completo de una Conversacion

1
Mobile crea o reutiliza una sesion (POST /api/ai/chats o sessionId existente).
2
Envia sessionId, message, context (userName, nationality, confirmedSearch, trip), limits y opcionalmente uiEvent.
3
API valida autenticacion, payload, cuota de sesion, lock de turno activo y moderacion de contenido.
4
Controller activa lock de turno (90s) y abre conexion SSE. Emite thinking de inmediato.
5
OpenAI Call 1 (gpt-5.4): recibe historial + contexto + herramientas disponibles. Decide intent y herramienta via function calling. Emite trace con la decision.
6
Backend ejecuta la herramienta seleccionada: busqueda en catalogo, disponibilidad real, detalle de hotel, planning o respuesta sobre resultados previos. Emite eventos de progreso (searching_catalog, searching_live, results_partial).
7
Si la herramienta es search_stays: corre el motor semantico, filtra inventario, arma carruseles y calcula scores. Emite results_final.
8
Si se necesita redactar: OpenAI Call 2 (gpt-5-mini o gpt-5.4 segun complejidad) genera texto conversacional en streaming. Emite kickoff_delta y luego chunks de delta.
9
Si hay cierre: genera mensaje de cierre streameable (closing_delta, assistant_closing).
10
API emite done, libera el lock de turno y persiste respuesta, plan_snapshot, inventory_snapshot y ui_snapshot en base de datos.
Turn lock: mientras se procesa un turno, cualquier otro request a la misma sesion devuelve 409 AI_CHAT_TURN_IN_PROGRESS. Mobile muestra el error al usuario y ofrece esperar o cancelar via POST /api/ai/chat/cancel.

6. Motor Semantico

insiderBack/src/modules/ai/ai.service.js — insiderBack/src/modules/ai/ai.semanticCatalog.js

El motor semantico es la capa que convierte texto libre en intents estructurados para guiar la busqueda y el ranking de hoteles. Corre en dos fases: primero detecta patrones via regex (sin costo de API), luego enriquece con modelo si hay ambiguedad.

Fase 1 — Extraccion deterministica (regex)

Cinco funciones corren sobre el mensaje del usuario antes de invocar ningun modelo:

FuncionDetectaEjemplos de triggers
detectSemanticViewIntent()Preferencia de vista"con vista al mar", "river view", "frente al rio"
detectSemanticGeoIntent()Relacion geografica con lugar"en Palermo", "cerca del Obelisco", "frente al agua"
detectSemanticAreaIntent()Tipo de zona buscada"buena zona", "zona tranquila", "en el centro"
detectSemanticQualityIntent()Nivel de calidad/precio"economico", "lujoso", "mejor relacion precio-calidad"
detectSemanticAreaTraits()Caracteristicas del area"tranquilo", "familiar", "zona de bares", "seguro"

Si alguna de estas detecciones produce un resultado con suficiente confianza, se evita la llamada al modelo semantico (ahorro de latencia y costo).

Fase 2 — Enriquecimiento por modelo

Cuando la extraccion deterministica es ambigua o incompleta, se invoca AI_MODEL_SEMANTIC_ENRICHMENT (default: gpt-5.4) con un schema JSON estricto. Timeout: 8000ms. El resultado se normaliza contra los vocabularios controlados antes de ser usado.

Catalogo semantico (ai.semanticCatalog.js)

Archivo de 1025 lineas con ~35 ciudades definidas con zonas, landmarks, aliases, coordenadas y traits. Version: 2026.03.enterprise.v1. Resuelve referencias locales sin llamar a ningun API externo.

Ciudades en el catalogo:

Buenos AiresDubai TokyoKyotoOsaka New YorkMiamiLos Angeles San FranciscoLas VegasOrlando ChicagoBostonWashington SeattleAustinNashville New OrleansHonoluluAtlanta DallasHoustonPhiladelphia PhoenixSan DiegoMexico City Rio de JaneiroSao PauloLima BogotaMedellinSantiago CartagenaPunta Cana YokohamaFukuokaSapporo

Cada ciudad tiene zones (barrios/distritos con coordenadas y radio) y landmarks (monumentos, estaciones, puertos) con sus alias en multiples idiomas. El web resolver cachea resoluciones externas por 30 dias con TTL configurable.

Regla de abstraccion: para queries como "tranquilo y caminable" o "buena zona", el sistema llena areaIntent y areaTraits primero. Solo inventa geoIntent o placeTargets si el usuario nombro explicitamente un lugar. Esto evita resultados geograficamente incorrectos.

7. Busqueda de Hoteles y Disponibilidad Real

insiderBack/src/services/assistantSearch.service.js

Catalog Discovery

Modo exploratorio. Puede buscar sin fechas, huespedes o nacionalidad completa. Sirve para mostrar opciones iniciales y orientar al usuario.

Execution mode: CATALOG_DISCOVERY

Live Availability

Requiere fechas, huespedes y nacionalidad. Si falta algo, interrumpe y pide los datos via UI estructurada. Consulta disponibilidad y precios reales.

Execution mode: LIVE_AVAILABILITY

Condiciones que fuerzan Live Availability

  • Frases en el mensaje: disponibilidad real, precio real, live availability, show live rates.
  • uiEvent === "LIVE_AVAILABILITY_ENABLE" enviado por mobile.
  • context.forceLiveAvailability === true.
  • Cuando la UI envia fechas y huespedes confirmados para una busqueda existente.

Scoring semantico

Cada hotel recibe un semanticScore calculado contra el intent del usuario. El score combina: match de traits de zona (semanticEvidence), match de nombre de hotel (candidateHotelNames), match de zona exacta, y seniority de evidencia.

  • Con viewIntent: se aplica si strongEvidence o semanticScore ≥ 24.
  • Con qualityIntent: se aplica si semanticScore ≥ 12.
  • Sin intent especial: cualquier score positivo activa el filtro semantico.

Carruseles (ai.carousels.js)

Los resultados se organizan en carruseles por categoria. Max 5 carruseles, 5 items cada uno. Si el inventario total es menor a 12 items, se muestra un unico carrusel de "Recomendaciones" para evitar repeticion. El precio se extrae normalizando multiples campos posibles del hotel (priceFromPerNight, priceFrom, nightlyRate, etc.).

8. Follow-Ups, Web Search y Fallback Cercano

Follow-ups sobre resultados previos

Si existe lastShownInventorySummary en el estado de la sesion, el backend puede responder preguntas sobre amenities, precios y comparaciones sin lanzar una busqueda nueva. El inventario raw se cachea por sessionId. El modulo ai.lastResultsFacts.js tiene un catalogo de features detectables por regex: pool, spa, gym, wifi, parking, breakfast, y mas.

Web Search

insiderBack/src/modules/ai/ai.webSearchPolicy.js

Se activa cuando...

  • El usuario pide fuentes, links, verificar, buscar online o Google.
  • Hechos actuales: hoy, ahora, latest, current, news.
  • Informacion variable: horarios, visa, restricciones, eventos, clima.

No se activa cuando...

  • Saludos o preguntas de capacidad del assistant.
  • Recomendaciones locales sobre resultados ya mostrados.
  • Preguntas que el assistant puede responder desde su conocimiento.

El modelo usado para web search es gpt-4o-search-preview. Las fuentes de competidores (Booking.com, Expedia, Hotels.com, Kayak, Agoda, Trivago, Airbnb) son filtradas automaticamente, salvo que el usuario las pida explicitamente. La politica de web search es geo_official_first.

Fallback Cercano

Si no hay hoteles en el destino exacto, el servicio expande geograficamente alrededor del anchor con tres radios progresivos:

5.000 m 15.000 m 35.000 m

El resultado se marca con metadata: nearbyFallbackApplied, nearbyFallbackCities y nearbyFallbackOriginalDestination. El assistant usa esta informacion para explicar que las opciones mostradas son cercanas al destino buscado, no exactamente en el.

9. Streaming SSE: Protocolo y Eventos

El servidor emite eventos SSE con el formato estandar: event: <tipo>\ndata: <JSON>\n\n. El controller es quien escribe directamente en res; el service recibe callbacks (onKickoffChunk, onClosingChunk, emitTrace, etc.) para mantener la separacion de capas.

Ciclo de eventos de un turno completo

Evento SSECuando se emiteData relevante
thinkingInmediato al recibir el mensaje. Antes de Call 1.{}
traceDespues de Call 1. Muestra la decision del modelo.{ code, label, destination, toolName, strategy }
kickoff_deltaChunks del mensaje inicial de busqueda (streaming temprano).{ content }
assistant_messageMensaje introductorio completo antes de resultados.{ content, nextProgressType }
searching_catalogInicio de busqueda en catalogo.{ destination, total }
searching_liveInicio de busqueda de disponibilidad real.{ destination, adults, children }
results_partialResultados parciales mientras sigue buscando.{ items[], total }
results_finalResultados finales con carruseles y semantica completa.{ carousels[], items[], semanticEvidence[] }
deltaChunks de la respuesta conversacional (Call 2 streaming).{ content }
closing_deltaChunks del mensaje de cierre.{ content }
assistant_closingMensaje de cierre completo con metadata.{ text, annotations[] }
doneFin del turno. Incluye texto completo persistido.{ text, sessionId, messageId }
errorError durante el procesamiento.{ message, code }

Diferencia Web vs. Mobile

Web (legacy)

bookingGPTFront/apps/web/src/services/aiAssistant.js

  • Usa fetch() directamente contra /api/ai/chat/stream.
  • Timeout idle: 30000ms.
  • Si el navegador no soporta streaming (ReadableStream), parsea el SSE completo al final.
  • Assistant.jsx maneja kickoff, progreso, resultados y cierre.
  • Modo mantenimiento: VITE_AI_CHAT_MAINTENANCE=true o localStorage.bookinggpt_ai_maintenance=1.

Mobile (principal)

bookingGPTFront/apps/mobile/src/services/aiAssistant.js

  • Intenta XMLHttpRequest primero (mejor soporte SSE en React Native).
  • Si XHR no esta disponible, cae a fetch().
  • Idle timeout: 30000ms entre chunks.
  • Soporte completo de cancelacion via /api/ai/chat/cancel.
  • Modo mantenimiento: EXPO_PUBLIC_AI_CHAT_MAINTENANCE=true.
  • AiAssistantContext.js orquesta todo el estado de UI.

AiAssistantContext.js

bookingGPTFront/apps/mobile/src/context/AiAssistantContext.js

Es el corazon del estado de IA en mobile. Expone el hook useAiAssistant() que todas las pantallas y componentes consumen. Responsabilidades:

  • Gestion de sesiones: crear, listar, cargar y eliminar sesiones.
  • Streaming SSE: recibe chunks y actualiza estado reactivamente. Distingue XHR de fetch segun disponibilidad.
  • Progreso de UI: mapea cada evento SSE a un label localizado (resolveStreamProgressLabel) y a una linea de trace (resolveStreamTraceLine).
  • Trace lines: acumula traceLines deduplicadas por clave code:label para mostrar el razonamiento interno al usuario.
  • Mensajes: adapta el formato del servidor al modelo interno de mensajes mobile (adaptServerMessages).
  • Inventario: detecta si un snapshot tiene datos de hoteles/homes y construye el inventario de UI (buildInventoryFromItems).
  • Carruseles: detecta si hay datos de carrusel validos (hasCarouselData).
  • Client guard timeout: AI_CLIENT_GUARD_TIMEOUT_MS = 30000ms como seguridad adicional del lado del cliente.
  • Localizacion: detecta locale y timezone del dispositivo (resolveLocale, resolveTimeZone).
  • Debug: EXPO_PUBLIC_DEBUG_AI=true habilita logs detallados en desarrollo.

Search Context Key (ai.searchContext.js)

Antes de ejecutar una busqueda, el sistema genera una clave de contexto que combina la firma del destino (ciudad, pais, lat/lng normalizados a 4 decimales) con el perfil semantico (star ratings, traits, zonas, landmarks). Si la clave es identica a la anterior, se evita relanzar la busqueda (deduplicacion).

Planner: applyPlanToState

La funcion applyPlanToState(state, plan) hace un merge profundo del plan del turno actual con el estado previo. Prioridades del merge:

  • Los valores null o undefined del plan entrante no sobreescriben el estado.
  • El destinationName se reconstruye desde city + state + country.
  • Las coordenadas (lat, lon) se normalizan a numeros flotantes.
  • El presupuesto (budget.min, budget.max) se normaliza a numeros.
  • Los adultos y ninos se normalizan y validan antes de guardar.

Funciones del state store

getDefaultState()Retorna el estado inicial limpio de una sesion. loadAssistantState()Carga el estado persistido de una sesion desde base de datos. saveAssistantState()Persiste el estado actualizado despues de cada turno.

12. Modulos de Soporte

ai.renderer.js

Convierte los resultados de busqueda y el estado semantico en payloads listos para ser consumidos por la app. Funciones principales:

  • renderAssistantPayload(): construye el payload completo del assistant con carruseles, inventario y metadata semantica.
  • buildDeterministicSemanticExplanationPlan(): genera el plan de explicacion semantica sin invocar el modelo (rapido).
  • getTopInventoryPicksByCategory(): selecciona los mejores hoteles por categoria para mostrar en carruseles.

ai.telemetry.js

Modulo de observabilidad y debug. Controlado por flags de entorno:

AI_EVENT_LOGSActiva logging de eventos en consola. AI_FILE_DEBUGEscribe debug detallado en archivo ./logs/ai-semantic-debug.txt. AI_FILE_DEBUG_PATHRuta del archivo de debug. AI_FILE_DEBUG_MAX_DEPTHProfundidad max de serializacion de objetos (default: 6). AI_FILE_DEBUG_MAX_ARRAY_ITEMSItems max en arrays serializados (default: 12). AI_FILE_DEBUG_MAX_STRING_LENGTHLargo max de strings (default: 1200 chars).

El debug se escribe de forma asincrona con cola de escritura para no bloquear el request.

ai.webSearchPolicy.js

Decide si una query debe activar web search. Evalua el mensaje del usuario contra una lista de patrones en multiples idiomas. Filtra automaticamente fuentes de competidores. Politica de fuentes: geo_official_first.

ai.lastResultsFacts.js

Permite responder preguntas sobre amenities de hoteles ya mostrados sin lanzar una busqueda nueva. Tiene un catalogo de features con deteccion por regex en espanol e ingles: pool/pileta, spa, gym/gimnasio, wifi, parking/estacionamiento, breakfast/desayuno, y mas.

ai.carousels.js

Agrupa el inventario de hoteles en carruseles por categoria para display en la app. Normaliza precios desde multiples campos posibles del hotel. Umbral para carrusel unico: menos de 12 items totales.

ai.policy.js

Aplica restricciones de negocio sobre el intent. Actualmente: si bookingFlowLocked es true, bloquea nuevas busquedas y redirige al usuario a ayuda general con aviso BOOKING_LOCKED.
